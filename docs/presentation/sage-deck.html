01 Biznesowo — po co to jest

Najskuteczniejsza metoda nauki, której nikt nie używa pod deadline

Profesjonaliści (dev / lekarz / prawnik) wiedzą, że powtórki działają — ale ręczne tworzenie fiszek jest tak drogie w czasie, że rezygnują z nich dokładnie wtedy, gdy są najbardziej potrzebne.

  • Insight: fiszki z AI przekroczyły próg „good enough" — kompromis „wolne ręczne vs. żadne" się odwrócił.
  • Wartość: próg wejścia spada z godzin do minut — wklej tekst, zaakceptuj karty, ucz się.
  • Persona: „upskilling professional" w oknie przed-deadline'owym z dużym backlogiem materiałów.

02 Produktowo — co realnie robi

Od surowego tekstu do talii, którą da się powtarzać

✶ Generacja AI

Wklejasz fragment — Sage drafuje kandydatów na fiszki Q/A z materiału.

✶ Review: accept / edit / reject

Decydujesz per-kandydat. Nic nie wchodzi do talii, dopóki nie zapiszesz; sesja przeżywa odświeżenie strony.

✶ Spaced repetition (FSRS)

Zapisane karty planowane algorytmem ts-fsrs v5 — powtórki w sam moment przed zapomnieniem.

✶ Konta i auth

Email/hasło: rejestracja, logowanie, reset hasła. Sesje cookie-based SSR (Supabase).

✶ Dwujęzyczność EN / PL

UI rozwiązywane server-side — bez migotania języka przy starcie.

✶ Pełne SSR

Wszystkie strony renderowane na serwerze; trasy chronione przez middleware.

04 Infrastruktura

Serverless na brzegu sieci, dane w Supabase

☁ Cloudflare Workers

Runtime workerd, adapter @astrojs/cloudflare v14 — auto-bindings IMAGES + SESSION. Worker sage-flashcards, live na *.workers.dev.

⛁ Supabase

Postgres + PostgREST + RLS. Migracje w supabase/migrations/ (flashcards + harmonogram FSRS), RLS na każdej tabeli.

⚷ Sekrety

astro:env/server → Worker secrets w produkcji (wrangler secret put). SUPABASE_*, LLM_API_KEY — nigdy w repo.

⚙ Dev & deploy

Lokalnie: Docker → Supabase + Mailpit, .dev.vars dla workerd. Deploy: build → wrangler deploy.

Mała skala, niski QPS, świadomie lean — MVP, który da się utrzymać po godzinach.

05 Repo — automatyzacja, review, testy

CI, piramida testów i trzy boty pilnujące repo

CI pipeline

lint + test + build na każdy push i PR do master.

Piramida testów

Unit (Vitest) · Integration (Vitest + Supabase, RLS) · E2E (Playwright + Mailpit) · Mutation (Stryker).

Workflow „10x"

plan → plan-review → implement → impl-review → archive. Każda zmiana ma ślad decyzyjny w context/.

Code-review bot
@sage/code-reviewer · Codex SDK

Ocenia diff każdego PR-a w 6 kryteriach → werdykt pass/fail + komentarz + etykieta + commit status. Re-run etykietą ai-cr:review.

ai-cr:passed ai-cr:failed
Release bot
release-please

Merge do master → Release PR z bumpem SemVer (z konwencjonalnych commitów) + CHANGELOG.md. Merge → tag vX.Y.Z + GitHub Release.

GitHub App token
Dependabot
npm + GitHub Actions

Otwiera grupowane PR-y z aktualizacjami zależności. Bezpieczne aktualizacje przewidziane do auto-merge po zielonym CI.

grouped

Izolacja sekretu (dwie fazy): producer pull_request bez sekretu (diff → artefakt) → consumer workflow_run z gałęzi domyślnej, z sekretem — PR-autor nie wykradnie OPENAI_API_KEY. Branch protection: PR + squash-only + linear history + wymagane checki. Conventional commits wymuszane dwustronnie (husky commitlint + check tytułu PR).
